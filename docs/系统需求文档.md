# SIGS 投诉即办系统需求文档

## 1. 项目概述

SIGS 投诉即办系统用于支持学生、教职工及相关用户提交事项工单，相关部门管理员进行办理、转办、回复和结果反馈，超级管理员进行系统配置、授权管理和数据统计分析。

系统目标包括：

- 支持用户便捷提交事项。
- 支持部门管理员按权限处理事项。
- 支持事项流转、回复、附件、满意度评价和典型问题发布。
- 支持统一身份认证登录。
- 支持统一待办通知推送。
- 支持中英文页面展示。
- 支持远程 MySQL 数据库统一存储。

## 2. 用户角色

### 2.1 普通用户

普通用户可以：

- 通过统一身份认证登录系统。
- 提交新的事项工单。
- 查看自己发起的事项。
- 查看事项办理进度、部门答复和附件。
- 在部门处理完成后确认问题是否解决。
- 对已解决事项进行满意度评价。
- 对未解决事项补充说明，并让同一工单重新进入处理流程。

### 2.2 部门管理员

部门管理员可以：

- 查看授权部门范围内的事项。
- 处理当前承办部门的待办事项。
- 回复事项处理结果。
- 上传处理附件。
- 将事项转办至其他部门。
- 查看事项详情、流转记录、提交人补充说明和满意度结果。
- 发布或取消发布典型问题。
- 查看授权范围内的数据统计分析。

### 2.3 超级管理员

超级管理员可以：

- 查看和处理全部事项。
- 管理事项领域配置。
- 管理部门配置。
- 配置中英文显示名称。
- 管理部门管理员授权。
- 将部门管理员升级为超级管理员。
- 查看全局数据统计分析。
- 管理本地账号。

## 3. 登录与身份认证

### 3.1 统一身份认证登录

系统支持通过清华 SIGS 统一身份认证登录。

功能要求：

- 未登录用户访问系统时，自动跳转统一身份认证系统。
- 无有效 token 时，前端清理本地旧身份缓存并直接进入后端 SSO 跳转入口。
- 登录成功后，系统保存 token 和用户基本信息。
- 登录过期或 token 失效时，清理本地认证信息并提示重新登录。

### 3.2 本地账号登录

系统保留本地账号密码登录入口。

功能要求：

- 支持本地账号登录。
- 支持首次登录强制修改密码。
- 支持超级管理员账号恢复和密码重置。

### 3.3 身份优先级

鉴权规则：

- 请求中携带 `Authorization: Bearer token` 时，优先使用 token 身份。
- 只有请求没有 token 时，才回退使用 session cookie。
- 避免浏览器残留旧管理员 session 导致普通用户被误判为管理员。

## 4. 普通用户端功能

### 4.1 提交事项

用户可以提交事项工单。

字段包括：

- 标题。
- 事项领域。
- 部门。
- 内容。
- 附件。
- 联系电话。
- 是否匿名。

功能要求：

- 表单领域和部门来自后台配置。
- 英文页面优先显示英文显示名。
- 提交时仍保存中文原始值，避免影响部门分派和统计。
- 支持附件上传。
- 附件类型限制包括 txt、docx、xlsx、pdf、png、jpg、jpeg、zip、avi、mp4。
- 单文件大小限制为 20MB。

### 4.2 我的事项

用户可以查看自己提交的事项列表。

功能要求：

- 显示全部事项。
- 按状态区分待处理和已完成。
- 点击事项进入详情页。

### 4.3 事项详情

用户可以查看事项详情。

展示内容包括：

- 事项标题。
- 提交时间。
- 提交部门。
- 当前状态。
- 用户提交内容。
- 用户上传附件。
- 部门答复。
- 回复附件。
- 办理进度时间线。
- 提交人补充说明。
- 满意度评价。

### 4.4 问题是否解决确认

部门管理员回复并标记完成后，提交人需要确认问题是否解决。

功能要求：

- 在已完成且已有部门回复的事项详情页显示“问题是否已解决？”模块。
- 用户选择“是，已解决”后，系统自动滚动到满意度评价区域。
- 用户选择“否，还未解决”后，显示补充说明文本框。
- 用户提交补充说明后，同一个工单重新进入待相关部门处理状态。
- 不创建新工单。
- 当前承办部门管理员收到通知和统一待办。

### 4.5 满意度评价

事项处理完成后，用户可以进行满意度评价。

功能要求：

- 支持 1 到 5 分评分。
- 支持填写评价说明。
- 每个事项只能评价一次。
- 评价提交后不可修改。
- 评价提交后完成提交人的统一待办。

## 5. 部门管理员端功能

### 5.1 事项处理列表

部门管理员可以查看授权范围内事项。

功能要求：

- 分为待相关部门处理和处理完成两个状态列表。
- 不显示手动刷新按钮。
- 数据加载时自动加时间戳避免缓存。
- 待处理事项不显示“处理部门答复”模块。
- 已回复事项显示处理部门答复。

### 5.2 事项详情弹窗

部门管理员点击事项后打开详情弹窗。

展示内容包括：

- 事项标题。
- 工单编号。
- 提交时间。
- 提交人。
- 联系方式。
- 事项内容。
- 附件。
- 当前状态。
- 提交人补充说明。
- 处理部门答复。
- 回复附件。
- 事项流转流程。
- 转办记录。
- 满意度调查结果。

### 5.3 回复事项

部门管理员可以对当前承办部门的事项提交处理结果。

功能要求：

- 只有当前承办部门管理员可以处理。
- 回复时可以上传附件。
- 回复提交后，事项状态更新为处理完成。
- 页面自动刷新详情和列表。
- 提交人收到站内通知和统一待办。
- 提交人需要确认问题是否解决并评价服务。

### 5.4 附件下载

功能要求：

- 用户和管理员均可在有权限的情况下下载附件。
- 附件下载接口支持 token query 参数，避免新窗口下载时丢失 Authorization header。
- 管理员和提交人均需通过权限校验后才能访问附件。

### 5.5 转办事项

部门管理员可以将事项转办给其他部门。

功能要求：

- 只能由当前承办部门管理员发起转办。
- 不能转办给当前承办部门。
- 转办后事项状态回到待处理。
- 当前承办部门变更为目标部门。
- 目标部门管理员收到站内通知和统一待办。
- 原部门管理员的统一待办被完成。
- 保留转办记录。

### 5.6 发布典型问题

部门管理员可以将已完成事项发布为典型问题。

功能要求：

- 只有处理完成事项可以发布。
- 支持取消发布。
- 典型问题页面展示公开事项和办理答复。

## 6. 超级管理员功能

### 6.1 配置管理

超级管理员可以维护表单配置。

配置内容包括：

- 事项领域。
- 部门。
- 部门类型。
- 是否启用。
- 英文页面显示名。

功能要求：

- 中文名称作为系统内部保存值。
- 英文显示名仅用于英文页面展示。
- 英文显示名为空时回退显示中文名称。
- 支持新增、编辑、删除、启用和禁用配置项。

### 6.2 授权管理

超级管理员可以管理部门管理员授权。

功能要求：

- 新增授权时，先选择授权部门，再根据部门搜索该部门人员。
- 人员搜索仅返回可授权范围内人员。
- 授权人员必须是在职人员。
- 教职员必须满足 `status = on_the_job`。
- 院外人员或 faculty 必须满足 `status = true`。
- `is_active` 必须为 1。
- 学生、离职、退休、死亡、状态异常人员不能授权。
- 支持一个人员管理多个部门。
- 支持启用/禁用授权。
- 支持编辑授权部门。
- 支持删除授权。
- 不再支持观察员角色。

### 6.3 升级超级管理员

超级管理员可以将部门管理员升级为超级管理员。

功能要求：

- 授权列表中提供升级按钮。
- 操作前需要二次确认。
- 升级后更新人员角色为 `super_admin`。
- 同步更新人员 `role_id`。
- 如果该人员有本地账号，同步更新本地账号角色。
- 删除该人员原有部门授权，避免超级管理员残留部门范围限制。
- 写入权限审计日志。

### 6.4 数据统计分析

管理员可以查看事项统计。

统计内容包括：

- 可见事项总数。
- 待处理事项数。
- 已完成事项数。
- 完成率。
- 状态分布。
- 部门分布。
- 领域分布。
- 满意度分布。

功能要求：

- 超级管理员查看全局统计。
- 部门管理员查看授权范围内统计。
- 统计范围包括当前部门、原始部门和历史转办经手部门。
- 无数据时显示空状态。
- 支持中英文显示。

## 7. 通知与统一待办

### 7.1 站内通知

系统支持站内通知。

触发场景包括：

- 新事项提交。
- 部门管理员回复。
- 事项转办。
- 提交人反馈未解决并补充说明。
- 管理员权限自动停用。

通知功能包括：

- 通知列表。
- 未读数量。
- 标记已读。
- 通知目标链接。

### 7.2 统一待办

系统支持向统一待办平台推送待办。

触发场景包括：

- 新事项提交后，推送给目标部门管理员。
- 部门管理员回复完成后，推送给提交人确认处理结果并评价。
- 事项转办后，推送给目标部门管理员。
- 提交人反馈未解决后，重新推送给当前承办部门管理员。
- 满意度评价完成后，完成提交人待办。
- 管理员处理或转办后，完成对应管理员待办。

待办幂等规则：

- 系统使用固定待办 ID 实现幂等覆盖。
- 待办 ID 格式为 `028d0a7edb_{工单ID}_{角色}_{人员ID}`。
- 重复推送同一个 ID 会更新同一条待办，而不是创建多条。
- 本地数据库目前不单独维护待办推送记录表。

## 8. 国际化功能

系统支持中文和英文页面。

功能要求：

- 支持中英文切换。
- 菜单、按钮、状态、提示、统计标题等页面文案支持国际化。
- 配置管理支持维护事项领域和部门的英文显示名。
- 英文页面优先显示英文名。
- 业务内部保存值仍使用中文原始名称。
- 下拉框、统计模块和详情页主要文案已完成英文适配。

## 9. 数据库与数据存储

### 9.1 数据库类型

系统统一使用远程 MySQL 数据库。

要求：

- SQLite 已弃用。
- 后端数据库模块统一使用 `server/db_mysql.js`。
- 旧 SQLite 相关代码和迁移脚本已移除。

### 9.2 核心业务表

主要表包括：

- `tickets`：事项工单。
- `replies`：部门回复。
- `attachments`：附件。
- `transfers`：转办记录。
- `ticket_followups`：提交人未解决补充说明。
- `notifications`：站内通知。
- `satisfaction_surveys`：满意度评价。
- `datahub_basic_persons`：同步人员基础信息。
- `departments`：部门配置。
- `form_options`：事项领域配置。
- `department_assignments`：部门管理员授权。
- `permission_audit_log`：权限操作审计。
- `roles`：角色。
- `permissions`：权限。
- `role_permissions`：角色权限关联。
- `users`：本地账号。

### 9.3 旧表清理

已清理的旧授权表：

- `department_admins`
- `department_admin_departments`

清理原则：

- 系统只使用 `department_assignments` 判断部门管理员授权。
- 旧授权表不再创建、不再读取、不再影响登录身份。

## 10. 人员数据同步

系统支持从 DataHub 同步人员基础数据。

功能要求：

- 支持按页同步人员数据。
- 每页数量可配置。
- 当前同步策略为每页 100 条。
- 当最后一页返回数量小于 100 时，视为同步完成。
- 同步后写入或更新 `datahub_basic_persons`。
- 同步运行记录写入 `datahub_basic_person_sync_runs`。
- 同步后自动检查离职人员，并停用其部门管理员授权。

当前已执行同步结果：

- 每页 100 条。
- 同步页数 313 页。
- 拉取总数 31,230 条。
- 写入/更新 31,230 条。
- 当前库内总人数 28,450。
- 最后一页 30 条，满足完成条件。

## 11. 权限控制

系统权限原则：

- 普通用户只能查看和操作自己提交的事项。
- 部门管理员只能处理当前承办部门事项。
- 部门管理员可以查看其授权部门、原始部门或历史转办经手部门相关事项。
- 超级管理员可以查看和处理全部事项。
- 附件下载必须通过事项权限校验。
- 授权管理和配置管理仅超级管理员可操作。
- 升级超级管理员仅超级管理员可操作。

## 14. 已知设计说明

### 14.1 统一待办推送记录

当前系统不在本地维护待办推送记录表，而是使用统一待办平台的固定 ID 实现幂等。

优点：

- 简单可靠。
- 重复推送不会生成多条待办。

限制：

- 本系统无法直接查询某条待办是否已成功推送。
- 需要通过日志或统一待办平台确认推送状态。

### 14.2 英文显示名

配置项英文显示名只用于前端展示，不改变业务存储值。

原因：

- 避免影响部门匹配。
- 避免影响授权范围。
- 避免影响统计口径。

### 14.3 同一工单返办

提交人反馈“未解决”时，不创建新工单，而是保留同一个工单。

原因：

- 保留完整历史上下文。
- 避免同一问题拆成多个工单。
- 便于部门继续处理和统计。
